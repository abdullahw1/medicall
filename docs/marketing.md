# MediCall Technical Narrative for Hackathon Judges

MediCall is an autonomous medication adherence system designed for elderly patients who may not use smartphone apps or wearable workflows reliably. The product assumption is intentionally simple: if we have a patient profile and a phone number, the system should run daily outreach without requiring user-side technology adoption. The core promise is that adherence risk should be detected and escalated before families and clinicians discover problems too late.

At a platform level, MediCall is implemented as a multi-agent pipeline where each component has a separate responsibility and each stage feeds the next stage with structured data. The system is architected around outbound calling, transcript interpretation, deterministic escalation, and proactive web-grounded safety context. In practice, this means the system does not only ask whether medication was taken, but also adapts next-day call context based on newly discovered FDA safety and recall information.

The current implementation follows the sponsor-driven architecture required by the project concept. Vapi serves as the voice infrastructure and transcript roundtrip surface, Guild.ai represents orchestration and scheduling control, InsForge powers persistence and outbound notifications, and TinyFish provides autonomous web monitoring for FDA safety updates. This split is not cosmetic. Each component corresponds to a specific product capability that would otherwise need separate custom infrastructure and significantly increase build risk during a hackathon timeline.

Technically, each sponsor is integrated using the mode that best matches its runtime role. Vapi is integrated through REST plus webhook callbacks: the backend places outbound calls using the Vapi `/call` API and ingests completion events via `POST /api/vapi-webhook`. Guild.ai is integrated through the Guild CLI and Agent SDK, with a coded orchestrator agent that drives the daily sequence against backend routes. InsForge is integrated via the `@insforge/sdk` package for alert fanout behavior, while TinyFish's FDA monitoring role is currently implemented through direct web-feed retrieval and matching logic in project services. This means the autonomy behavior is live and observable even where a sponsor-specific SDK is not the transport layer.

From an execution-flow perspective, MediCall operates as a scheduled autonomy system that runs daily and weekly loops. The daily cycle begins with medication safety intelligence gathering, then continues into outbound patient contact, transcript classification, persistence, and conditional escalation. The weekly cycle compiles plain-English summaries for caregiver and clinician visibility. Together, these loops create both immediate intervention behavior and longitudinal reporting behavior, which is important for judge evaluation because it demonstrates day-zero utility and ongoing care value.

The nightly safety stage checks FDA recall and alert feeds and attempts to match recall language against each patient medication list. These matched items become contextual inputs for the next morning call stage. During call execution, the voice agent asks adherence and wellbeing questions and then classifies outcomes into a constrained status set. The canonical statuses are took_meds, missed_meds, no_answer, and concern. This normalized status model is intentionally narrow so alert logic remains auditable and judges can trace exactly why escalations happen.

Escalation behavior is deterministic and implemented as explicit business logic instead of probabilistic prompting. The current trigger conditions are concern status from a call, two or more recent missed doses, and three consecutive no-answer outcomes. When any condition is met, notification fanout is sent to caregiver and doctor contacts, and the call result is persisted with alert_sent = true. This deterministic approach is important in healthcare-adjacent workflows because it allows reproducibility, debugging, and confidence during demo evaluation.

The backend is a TypeScript service exposing API endpoints for health checks, patient retrieval, call result retrieval and ingestion, TinyFish FDA matching, and weekly summary generation. The active implementation map in this repository uses src/types.ts for shared schemas, src/routes/api.ts for request handling, src/services/alerts.ts for deterministic alert logic, src/services/tinyfish.ts for FDA fetch and matching logic, src/services/insforge.ts for notification integration, src/services/summary.ts for weekly narrative generation, and src/store.ts for current in-memory seed and retrieval behavior.

Data contracts are intentionally strict and shared across the build so teammates can work in parallel without integration drift. Patient records include identity and communication fields plus medication list and local call scheduling metadata. Call result records include call identity, patient identity, timestamp, normalized status, transcript text, extracted flags, embedded FDA alert context, and escalation state. This contract-first approach allows orchestration, backend, and dashboard layers to evolve independently while still remaining wire-compatible.

On the frontend side, the dashboard has been rebuilt as an operations-oriented care interface rather than a generic CRUD panel. The current screen is designed to communicate system health, call activity, adherence performance, escalations, patient roster status, and agent pipeline behavior at a glance. It includes manual controls for running FDA match checks and triggering live outbound calls, and it hydrates from live API responses on a refresh cadence. The UI intentionally emphasizes timeline evidence, transcript snippets, and escalation chips so judges can immediately connect architecture claims to visible outcomes.

The current project posture favors live-data integrity over synthetic continuity in judge-facing flows. If FDA feed retrieval fails or live telephony cannot be initiated, the API surfaces explicit errors rather than manufacturing substitute medical events. This keeps evidence trails honest and ensures dashboard state corresponds to real upstream behavior.

From an autonomy framing perspective, MediCall demonstrates two distinct layers that judges often ask teams to separate. The first layer is scheduled autonomy, where daily and weekly pipelines run without manual initiation. The second layer is web-grounded autonomy, where live web-sourced safety updates alter future behavior without human rewriting of prompts or rules. This distinction helps position MediCall as adaptive infrastructure rather than only a scripted reminder bot.

For evaluation and storytelling, the strongest demonstration path is to show a complete loop that starts with a patient context, executes a call, records a transcript-derived status, triggers or suppresses alerts based on deterministic thresholds, and then surfaces that result in the dashboard and weekly summary narrative. In this loop, each visible UI state has a corresponding backend event and each backend event maps to explicit business logic. This traceability is usually persuasive for hackathon judges because it proves implementation depth rather than only concept polish.

The current build is intentionally scoped to maximize shipping reliability while keeping architecture extensible. Immediate next evolutions include production-grade persistence instead of in-memory stores, hardened credential and compliance posture, richer pharmacology Q&A routing, and deeper report analytics. Even in its present state, the system already demonstrates sponsor-integrated orchestration, structured data contracts, autonomous outbound intervention, deterministic escalation logic, and a coherent operator interface suitable for live judging sessions.
